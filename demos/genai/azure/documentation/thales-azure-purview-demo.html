How the governed protection flow works

The workflow begins at the far left with Microsoft Purview discovery and classification. From there it branches into file and database patterns. File content uses Azure AI Language PII detection for precise in-content detection before Thales protection is applied. SQL uses Purview’s column classifications directly. Protected output becomes the approved source for vector stores, analytics, and AI pipelines.

File-driven pattern

Purview identifies which blobs or ADLS files are sensitive. The application then streams those exact assets from Azure Storage, uses the Azure AI Language PII detection API to identify precise PII in the content, and applies Thales protection before writing governed output back to storage.

SQL-driven pattern

Purview identifies which tables and columns contain sensitive data. The application protects only those columns, producing a governed SQL target table for analytics, AI preparation, or downstream sharing.
