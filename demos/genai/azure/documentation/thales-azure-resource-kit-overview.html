What the resource kit covers

This resource kit is broader than a single demo. It is a set of practical patterns that show how teams can protect sensitive data before it is used in prompt workflows, retrieval pipelines, analytics extracts, and governed data movement.

Customers can use it to tell a complete story: detect sensitive values with Azure AI Language PII detection where needed, use Purview to identify governed assets, and apply Thales protection before those assets are reused in downstream systems.

Prompt protection Protect prompt content before Azure OpenAI requests and inspect output content again for sensitive values.
File-to-file protection Detect PII in local files and write protected output for batch preparation and downstream reuse.
Governed processing Use Microsoft Purview to decide which storage assets and SQL columns should be processed.

Four solution patterns in the resource kit

The samples cover different entry points, but they work together as one narrative. Teams can start with a simple local file-to-file example, move to prompt protection, and then expand into Purview-driven governed workflows for storage and Azure SQL.

1. Non-Purview local file-to-file batch flow

The local batch processor reads files from an input directory, uses Azure AI Language PII detection to identify sensitive values, protects those values with Thales CipherTrust, and writes the protected output to a local destination directory.

local input pii detection api protected output

2. Prompt protection for Azure OpenAI

The prompt example protects prompt content before it is sent to Azure OpenAI, demonstrates response handling, and optionally checks returned content again for sensitive values. This is useful for showing application-layer protection around prompt workflows.

prompt workflow azure openai output inspection

3. Purview-driven storage protection

Purview identifies sensitive storage assets. The application reads those same blobs or ADLS files directly from Azure Storage, uses Azure AI Language PII detection for fine-grained in-content detection, and streams protected output back to a governed target path.

purview blob / adls streaming

4. Purview-driven Azure SQL protection

Purview identifies sensitive tables and columns. The application protects only the classified columns and writes governed output to a protected target table, creating a strong structured-data preparation path for analytics and AI.

azure sql column targeting structured data
Together, these patterns let customers start simple and then move toward a stronger governed architecture without abandoning the same Thales protection model.

How the kit supports vector DB and fine-tuning preparation

One of the strongest use cases for the resource kit is preparing data before it enters a vector database or a fine-tuning workflow. The most direct path begins with file-based content and uses Azure AI Language PII detection to locate sensitive values before Thales protection is applied.

Vector database preparation

The file-to-file path is a practical way to create a protected corpus before chunking, embedding, and indexing. The protected file becomes the approved source for vector-store ingestion rather than the original raw file.

Fine-tuning preparation

The same pattern can prepare training or tuning data by reducing exposure to raw personal data while preserving the broader structure and utility of the content.
