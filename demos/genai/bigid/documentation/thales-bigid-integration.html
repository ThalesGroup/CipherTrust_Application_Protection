How the BigID-governed protection flow works

The workflow begins with BigID catalog results and then branches into unstructured and structured paths. Unstructured assets are resolved from the right storage system, converted to text, inspected with Azure AI Language PII detection, and protected with Thales. Structured assets use BigID field classification directly to target sensitive SQL columns. The protected output becomes the approved source for vector stores, analytics, and governed AI use.

Unstructured path

BigID identifies which files or object-store assets are sensitive. The application resolves those exact assets, converts them to text, finds exact PII spans with Azure AI Language, and applies Thales protection before writing new text-based artifacts to the configured destination.

Structured path

BigID identifies the sensitive table fields. The application maps those fields to JDBC-accessible SQL columns, protects only those values, and writes a governed target table for analytics, AI preparation, or downstream sharing.

How unstructured parsing and review artifacts work

The BigID sample is designed for the common AI-preparation use case where the end result should be text or JSON rather than the original document format. That means binary documents are intentionally normalized into text before sensitive-value detection and protection occur.

PDF

PDF content is extracted into plain text with PDFBox so Azure AI Language can inspect readable content instead of raw binary bytes.

Word

Both `.doc` and `.docx` are normalized into paragraph text before PII detection and protection are applied.

Excel

`.xls` and `.xlsx` content is converted into CSV-like text rows so spreadsheet data can flow into the same text-based detection and protection path.

Text and CSV

Text-like content skips binary parsing and moves directly into the span-detection and protection stage.

The original local source file is never overwritten. Remote source objects are never overwritten either. The workflow writes new extracted, protected, and findings artifacts to the configured target location.

Why this approach fits AI pipelines

  • Vector databases and RAG pipelines usually want text, not binary documents.
  • Review artifacts make it easier to validate extraction quality and false-positive behavior.
  • One normalized text path simplifies policy enforcement across many document types.
  • Temp-file staging is limited to extraction only and the temp file is deleted immediately afterward.

Cloud sources, operating model, and customer value

One of the strongest parts of the BigID story is that discovery and processing do not have to be tied to one storage platform. If BigID catalog points to Azure Blob, S3, or GCS, the same protection flow can be applied as long as the matching source settings are configured.

Azure Blob and ADLS

When BigID returns an Azure-style path, the application parses container and blob names, downloads the object temporarily, converts it to text, protects the sensitive spans, and writes protected output to a target container and prefix.

Amazon S3

When BigID returns an S3-style path, the application parses bucket and object key, downloads the object, converts it to text, protects the sensitive spans, and writes output to the configured target bucket and prefix.

Google Cloud Storage

When BigID returns a GCS-style path, the application parses bucket and object name, downloads the object, converts it to text, protects the sensitive spans, and writes output to the configured target bucket and prefix.

Auto Mode

In `source.mode=auto`, the application tries local resolution first and then any configured remote stores in order: Azure Blob, S3, and GCS.

What operators need to provide

  • A BigID path format that the parser recognizes.
  • The right `source.mode` and provider-specific properties.
  • Credentials and target output settings for the chosen object store.
  • A supported file extension in `source.supportedExtensions`.

Recommended first run

  • Enable `bigid.debug.dumpAssets=true` and `bigid.debug.only=true`.
  • Inspect whether BigID asset paths parse into the expected Azure, S3, or GCS bucket and object values.
  • Turn on extracted text and findings JSON output for the first protection run.
  • Validate the protected text before connecting it to a vector database or search index.
