Why this matters

Databricks teams want to protect sensitive fields without forcing every notebook, dashboard, pipeline, and SQL model to implement custom security logic. The practical answer is to move protect and reveal behavior into a governed layer that the data platform can call consistently.

Thales provides the policy and key control plane, while Databricks provides the execution and analytics plane. The result is a scalable model for tokenization, detokenization, masking, and controlled reveal that works across data engineering, data science, and business analytics workloads.

How the Databricks architecture works

Databricks workloads call a governed function layer. That function layer invokes Thales CRDP for protect and reveal operations, while CipherTrust Manager remains the policy and key control plane. To reduce latency, CRDP should be deployed as close as possible to Databricks in Azure.

Recommended pattern

Use Unity Catalog functions and views as the common interface for governed SQL, BI, and pipeline workloads.

Latency model

The data path between Databricks and CRDP matters more than the small key and policy payload size between CRDP and CipherTrust Manager.

Operational split

CRDP acts as the high-throughput execution service, while CipherTrust Manager remains the source of truth for control and governance.

Why this architecture is practical

  • Protects data at the analytics access path instead of pushing custom logic into every application.
  • Supports SQL Warehouse, BI tools, engineering notebooks, Streaming, and Lakeflow or DLT with one governed function layer.
  • Lets security teams retain centralized policy and key ownership while data teams continue using familiar Databricks patterns.

Governed access and secure reveal patterns

The same input token should not necessarily produce the same reveal result for every user. For example, one user may see masked output while another sees full cleartext based on Thales user set membership and access policy. That is why reveal should be tied to Databricks session identity and exposed through governed functions and views.

Recommended security model Use SQL views to call the reveal function and inject session_user() or current_user() so the user never supplies a spoofable reveal identity.
CREATE OR REPLACE FUNCTION main.security.thales_crdp_scalar_by_column_with_user( value STRING, mode STRING, datatype STRING, column_name STRING, reveal_user STRING ) RETURNS STRING LANGUAGE PYTHON AS $$ from thales_databricks_udf.crdp_udfs import thales_crdp_python_function_bulk return thales_crdp_python_function_bulk( [value], mode, datatype, column_name=column_name, reveal_user=reveal_user )[0] $$;
CREATE OR REPLACE VIEW main.security.v_customer_email_reveal AS SELECT customer_id, main.security.thales_crdp_scalar_by_column_with_user( email_token, 'reveal', 'char', 'email', session_user() ) AS email FROM main.raw.customer_tokens;
Why reveal is different

Tokenization can be deterministic if the policy is deterministic, but detokenization can be non-deterministic when masking or cleartext depends on access policy and user-set membership.

BI and view advantage

Business intelligence tools can query the view like a normal database object while the reveal identity is injected by the Databricks engine.

Operational benefit

The same secure-view pattern can be reused in downstream views, semantic layers, and role-specific reporting datasets.

Performance and pipeline support

Bulk protect and reveal CRDP v1 bulk APIs are used to reduce round trips and improve throughput for Databricks ETL, batch processing, and Streaming workloads.
Batching by profile Batching is performed per resolved column and profile, not across mixed-profile values in a full row, because CRDP v1 bulk requests require one policy per call.
Optional reveal cache Executor-local reveal caching can be enabled for repeated detokenization workloads, with bounded size and explicit flags.
For Databricks Streaming and Lakeflow or DLT, the preferred implementation model is Unity Catalog SQL or Python functions. The same governed logic can be reused in batch jobs, SQL Warehouse, dashboards, and pipelines without maintaining separate code paths.

Implementation path and customer value

A practical rollout starts with governed functions and secure views, then expands into compute, Streaming, and Lakeflow or DLT as adoption grows. The same architecture lets security teams enforce policy centrally while giving data teams a familiar Databricks-native experience.

Phase 1

Publish Unity Catalog SQL/Python functions and secure reveal views for SQL Warehouse, analysts, and BI tools.

Phase 2

Add compute-cluster Java UDFs where engineering teams need notebook-oriented Spark integrations.

Phase 3

Reuse the same Unity Catalog functions inside Structured Streaming and Lakeflow or DLT pipelines for high-volume ETL.

Phase 4

Tune batch size, policy mappings, reveal caching, and Azure placement based on workload and policy behavior.

What customers gain

  • Governed tokenization and detokenization in Databricks without breaking SQL and analytics workflows.
  • One protection model for SQL Warehouse, BI, ETL, Streaming, and Lakeflow or DLT.
  • Centralized Thales control over keys, policy, access, and masking behavior.
  • Practical Azure deployment model with CRDP close to data and Databricks.
Thales brings the security control plane. Databricks provides the analytics plane. Together they enable scalable, governed protection for modern data and AI workflows.
