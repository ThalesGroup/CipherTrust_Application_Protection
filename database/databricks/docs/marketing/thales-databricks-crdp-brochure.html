How the Databricks architecture works

Databricks workloads call a governed function layer. That function layer invokes Thales CRDP for protect and reveal operations, while CipherTrust Manager remains the policy and key control plane.

Access model

Analysts, BI tools, notebooks, pipelines, and engineering jobs all enter through a governed Databricks access layer rather than embedding custom security logic in every workload.

Execution choices

Unity Catalog SQL/Python functions support SQL Warehouse and governed query use cases, while Java Spark UDFs support compute-cluster ETL and notebook-oriented Spark execution.

Control plane separation

CRDP performs runtime protect and reveal operations, while CipherTrust Manager remains the authority for keys, policies, user sets, and reveal rules.

Why this architecture is attractive

  • Lets data teams keep using Databricks-native tools and query patterns.
  • Lets security teams retain centralized Thales governance over reveal behavior.
  • Provides one practical architecture that can cover analytics, ETL, streaming, and governed access.

Optimization and scale options

The solution is not limited to a single execution pattern. It includes several optimization levers so organizations can tune for throughput, efficiency, and operational simplicity based on workload type.

Bulk API leverage Use bulk protect and reveal patterns where the source data shape supports them. This reduces request overhead and improves throughput for high-volume workloads.
Flexible tuning controls Shape grouped request size, CRDP request size, and Spark work distribution using grouping controls, partition counts, and batch-size settings.
Spark executor parallelism Take advantage of multiple executors and partitioned work so Databricks can keep more independent CRDP operations in flight.
Reveal caching options Optional executor-local reveal caching can help repeated detokenization workloads where the same protected values are revealed often.
Deployment tuning Scale CRDP vertically or horizontally depending on whether the goal is a simple benchmark baseline or a more resilient production architecture.
Shared service model Support a shared CRDP environment that can serve multiple Databricks jobs when aggregate concurrency and autoscaling behavior have been validated.
A strong production pattern is to combine Databricks executor parallelism, bulk API usage where applicable, and a right-sized CRDP deployment backed by observability and autoscaling.
